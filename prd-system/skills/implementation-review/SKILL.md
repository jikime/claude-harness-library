# 구현 리뷰 매뉴얼 (implementation-review)

목적: 구현 코드가 **스펙대로인지**를 외부 신호(테스트·스펙 대조)에 묶어 회의적으로 검증한다. 생성과 평가를 분리한다 — 구현 에이전트의 자기보고를 그대로 믿지 않는다.

## 시작 전 — 그라운딩 프로브
`TASKS.md`·`PRD.md`·`ARCHITECTURE.md`를 정독해 합격 기준을 먼저 고정한다. 그 다음 실제 코드를 read-only로 탐색한다. 가능하면 `TASKS.md`의 Commands로 테스트/빌드/린트를 실제 실행해 결과를 근거로 삼는다.

## 점검 항목
1. **태스크 충족**: T-### 각각이 실제로 구현됐는가. Verification(테스트/명령)이 실제로 통과하는가.
2. **추적성**: PRD FR-### → ARCH 컴포넌트 → TASK → 코드로 이어지는가. 구현됐지만 스펙에 없는 것(스코프 크립), 스펙에 있지만 누락된 것 양방향 점검.
3. **Acceptance Criteria**: PRD의 AC(샘플 입출력 포함)를 코드가 만족하는가.
4. **Boundary 준수**: TASKS의 🚫 위반(시크릿 커밋, prod 설정, node_modules 편집 등)이 있는가. ⚠️ 항목이 사람 검토 없이 처리됐는가.
5. **보안 (스펙 정합 한정)**: PRD §13에 명시된 보안 요구(인증/권한/입력검증/Rate Limit)가 **스펙대로 구현됐는가**만 본다. 일반 취약점 사냥·심층 보안 점검은 이 리뷰의 범위가 아니다 → **`pre-deploy-gate` 하네스로 위임**(중복 방지).
6. **NFR**: ARCHITECTURE §6의 실현 방안이 코드에 반영됐는가(가능한 범위).

## 출력 형식 — `implementation-review.md`
```
# Implementation Review: {제품명}

## 요약 — 권고 상태: 통과 권고 / 조건부 / 미통과 (판정은 권고이며 자동 merge·배포 안 함)

## 태스크별 결과
| Task | 상태(통과/부분/탈락) | 근거(실행한 테스트·파일) | 비고 |

## 추적성 갭
- 누락(스펙엔 있으나 미구현): …
- 스코프 크립(코드엔 있으나 스펙 없음): …

## Boundary 위반 / 보안 이슈
- (있으면 심각도와 함께)

## 권고 (사람 승인 필요 항목)
- merge/배포 전 사람이 확인해야 할 것
```

## 원칙
- 판정을 **외부 신호**에 묶는다: 실제 테스트 실행 결과, 스펙 항목 대조, boundary 체크. 순수 인상비평 금지.
- 회의적으로 본다: "통과"는 근거(실행 로그·파일 경로)가 있을 때만.
- **merge·배포·삭제는 실행하지 않는다**. 권고만 내고 사람 승인 게이트로 넘긴다.

## 품질 체크
- [ ] 모든 T-###에 근거 있는 상태가 매겨졌는가
- [ ] 추적성 갭을 양방향으로 점검했는가
- [ ] boundary 위반·**스펙 명시 보안 요구(§13) 정합**을 점검했는가 (심층 보안은 pre-deploy-gate로 위임)
- [ ] 가능한 경우 실제 테스트/빌드를 실행해 근거로 삼았는가
- [ ] 위험 행동(merge/배포)을 자동 실행하지 않고 권고로 남겼는가
